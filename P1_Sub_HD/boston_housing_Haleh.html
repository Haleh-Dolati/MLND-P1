Implementation: Define a Performance Metric¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 is no better than a model that always predicts the mean of the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is arbitrarily worse than one that always predicts the mean of the target variable.

Question 3 - Training and Testing¶

  • What is the benefit to splitting a dataset into some ratio of training and testing subsets for a learning algorithm?

Hint: Think about how overfitting or underfitting is contingent upon how splits on data is done.

Answer:
When it comes to training and testing the model, we have two choices:

  1. Using the entire dataset for training: in this case, we use the entire dataset for training the model. Then, a subset of the same dataset (which we already used for training the model) is used to test and evaluate the model.
  2. Split the data into test and training: In this case, we use a certain amount of data (usually 80%) for training the model. Then we use the rest of the data (usually 20%) for testing and evaluating the model.

The problem with the first approach is the test result wont tell us how our model would do for new unseen data. In other word, we are choosing a model based on its performance on the same data that we build that model based on! It should perform well. This will result in a problem that is known as over-fitting. Over-fitting is when the model is not generalized.
The second approach will let us to introduce the test data to the model after training, So we can test the model using unseen data.

Question 8 - Cross-Validation¶

  • What is the k-fold cross-validation training technique?

  • What benefit does this technique provide for grid search when optimizing a model?

Hint: When explaining the k-fold cross validation technique, be sure to touch upon what 'k' is, how the dataset is split into different parts for training and testing and the number of times it is run based on the 'k' value.

When thinking about how k-fold cross validation helps grid search, think about the main drawbacks of grid search which are hinged upon using a particular subset of data for training or testing and how k-fold cv could help alleviate that. You can refer to the docs for your answer.

Answer:
Dividing our data to training and test helps us to test our model and its performance with an unseen set of data. While this can help us to see how our model is doing, it gives us some some limitations. For example we can test our model only once! Instead, we can divide our training data to K equal bins and each time use k-1 of them for training and 1 for validating. We repeat this process K times so each bean is used one and only 1 time for validating and K-1 times for training. At the end, by taking the average of the scores of the validations we will get a single realistic validation score. These splits are called folds and K-folds mean we slit the data into K bins.

We then iterate through the k chunks one by one and use the current chunk for model validation and the remaining k-1 chunks for training. What we end up with are k trained and evaluated models and we have used the entire dataset for validation. Averaging the validation scores gives us a single validation score. It is more reliable than if we had used only one split. When a K-fold cross validation is combined with Grid Search, we can choose a model and its tuned parameters that perform the highest. It means we can find the best parameters for the model because there is less variance in performance estimate. Also in grid search because we are use the training-validation split so many times to different parameters combination, we may overfit the model. This happens because we try different combination and it might by chance find a point that gives us a really good performance. With K-Fold, since we are taking the average of K performances, this effect is minimized.

from sklearn.tree import DecisionTreeRegressor
from sklearn.metrics import make_scorer
from sklearn.grid_search import GridSearchCV

def fit_model(X, y):
    """ Performs grid search over the 'max_depth' parameter for a 
        decision tree regressor trained on the input data [X, y]. """
    
    # Create cross-validation sets from the training data
    # sklearn version 0.18: ShuffleSplit(n_splits=10, test_size=0.1, train_size=None, random_state=None)
    # sklearn versiin 0.17: ShuffleSplit(n, n_iter=10, test_size=0.1, train_size=None, random_state=None)
    cv_sets = ShuffleSplit(X.shape[0], n_iter = 10, test_size = 0.20, random_state = 0)

    # TODO: Create a decision tree regressor object
    regressor = DecisionTreeRegressor()

    # TODO: Create a dictionary for the parameter 'max_depth' with a range from 1 to 10
    params = {'max_depth': list(range(1,11))}

    # TODO: Transform 'performance_metric' into a scoring function using 'make_scorer' 
    scoring_fnc = make_scorer(performance_metric)

    # TODO: Create the grid search cv object --> GridSearchCV()
    # Make sure to include the right parameters in the object:
    # (estimator, param_grid, scoring, cv) which have values 'regressor', 'params', 'scoring_fnc', and 'cv_sets' respectively.
    grid = GridSearchCV(regressor, params, scoring = scoring_fnc, cv = cv_sets)

    # Fit the grid search object to the data to compute the optimal model
    grid = grid.fit(X, y)

    # Return the optimal model after fitting the data
    return grid.best_estimator_

Answer:

This model, as is, cannot be trusted and be used in a real-world setting today. Although it has been adjusted for the inflation, that is not the only reason that can affect the price of a house today compared to 40 years ago. Peoples perception of a neighborhood, change in lifestyle, gentrification, or change in education system, all can affect the price and cannot be captured by those 3 variables (number of bedrooms, Poverty in neighborhood, and student/teacher ratio). However, there are ways to improve the results:
1- Collecting more data for the past 40 years and add it to the dataset.
2- Using more features such as:

  • Square footage of the house
  • Yard size
  • Accessibility to public transit (walking distance, type of PT)
  • Proximity to major employment centers
  • Crime rate in the neighborhood

Also it worth to mention this model cannot be used for other cities. The buyers and sellers behavior and preferences are totally different on Columbus (Ohio), Palo Alto (California), and Dallas (Texas). So this model cannot be used in other cities or rural areas. In addition, because this model was built based few variables, it may give us not so accurate estimate even for houses in the same neighborhood.
Another problem is the sensitivity analysis shows this model is not well generalized. The minimum and maximum of the predicted price for the same client when we run the model 10 times, has almost 70,000 dollar difference.
